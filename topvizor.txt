Введение в API
Последнее обновление: 06.11.2020

URL запроса для обращения к серверу API
Все запросы к API должны направляться на https://api.topvisor.com с помощью метода POST.
URL запроса должен включать в себя следующие данные:

<!-- https://api.topvisor.com/{версия}/{формат}/{оператор действия}/{имя сервиса}[/{имя метода}] -->
HTML
Версия API (v2);
Формат возвращаемых данных (поддерживается только JSON);
Оператор действия
Имя сервиса
Имя метода (необязательно)
Заголовок: Content-type (необязательно, рекомендуется "application/json");
Заголовок: User-Id;
Заголовок: Authorization.
Пример URL запроса:

<!-- https://api.topvisor.com/{версия}/{формат}/{оператор действия}/{имя сервиса}[/{метод}] -->
https://api.topvisor.com/v2/json/get/users_2/profile/avatar
Markup

Доступ и авторизация
Для работы с API необходимо получить API Key.
Заголовки запроса для авторизации:

User-Id:		%USER_ID%
Authorization:	bearer %USER_API_KEY%
HTTP

Формат передачи данных
Все данные должны передаваться в кодировке utf-8.
Данные могут передаваться в теле запроса двумя способами: JSON или form-data.

Примеры заголовка запроса, устанавливающего тип передаваемых данных:

Content-type: application/json
HTTP
Content-type: ЛЮБОЙ_ДРУГОЙ // например x-www-form-urlencoded (не рекомендуется)
HTTP

Формат ответа
Ответ приходит в кодировке utf-8.
Поддерживается только один формат ответа: JSON.

{
"result": /* результат */,	"nextOffset": /* число (может отсутствовать) */,
"total": /* число (может отсутствовать) */,
"errors":[/* массив ошибок (может отсутствовать) */],
"messages":[/* массив строк с информационными сообщениями (может отсутствовать) */]
}
JSON
Подробнее о nextOffset и total читайте на странице постраничная выборка.
Подробнее об errors читайте на странице обработка ошибок.


Ограничение на использование API
В одном запросе на получение данных можно запросить не более 10 000 результатов
С одного IP не более 5 одновременных запросов
Для одного User-Id не более 5 одновременных запросов
При превышении ограничения API вернет ошибку с кодом 429.


Объекты данных
API позволяет получать, изменять и удалять объекты различных типов. Каждый сервис работает со своим типом данных. Перечень всех сервисов API и их объектов описан в разделе Сервисы.

Так к примеру сервис "Ключевые фразы" позволяет управлять такими объектами как "Ключевые фразы", "Группы" и "Папки".

Каждый тип данных имеет свой набор характеристик. Пример типа данных Ключевые фразы

Получение данных: оператор get
Последнее обновление: 06.11.2020
Для получения объектов и других типов данных используется оператор get.

POST /v2/json/get/{service}/{method}
HTTP
Для указания получаемых полей объектов, фильтрации, сортировки или порционного получения данных используйте соответствующие параметры запроса.

В качестве result будет возвращен массив объектов или значений в зависимости от метода.
В случае ошибки выполнения запроса, result будет равен null

Добавление объекта: метод add
Последнее обновление: 06.11.2020
Для добавления объектов используется оператор add.

POST /v2/json/add/{service}/{method}
HTTP
В качестве result будет возвращен идентификатор добавленного объекта.
Некоторые методы, добавляющие за раз несколько объектов, возвращают в качестве result массив их идентификаторов.
Некоторые методы позволяют в качестве result возвращать массив добавленных объектов, если передан параметр fields.
Некоторые методы в качестве result возвращают количество добавленных результатов.
В случае ошибки выполнения запроса, result будет равен null

Редактирование объекта / выполнение операции: оператор edit
Последнее обновление: 06.11.2020
Для редактирования объектов используется оператор edit.

POST /v2/json/edit/{service}/{method}
HTTP
При редактировании обязательным параметром является filters или id.

В качестве result будет возвращено количество затронутых объектов.
Некоторые методы позволяют возвращать в качестве result массив изменяемых объектов (даже если не все из них были изменены), если передан параметр fields.
В случае ошибки выполнения запроса, result будет равен null

Удаление объекта: оператор del
Последнее обновление: 06.11.2020
Для удаления объектов используется оператор del.

POST /v2/json/del/{service}/{method}
HTTP
При удалении обязательным параметром является filters или id.

В качестве result будет возвращено количество удаленных объектов.
В случае ошибки выполнения запроса. result будет равен null

Постраничная выборка (параметры limit и offset)
Последнее обновление: 06.11.2020
Данные параметры предназначены только для операций чтения, т.е. для работы с оператором get
Результаты любой выборки из API принято называть страницей результатов.
Каждая страница не может включать больше 10 000 результатов.
Размер страницы определяется параметром limit и по умолчанию равен 10 000.
Количество пропускаемых элементов при генерации страницы определяется параметром offset.

Если в результате выборки были возвращены не все результаты, то в ответе будет возвращены дополнительные параметры:

nextOffset - значение offset для выборки следующией страницы
total - общее количество результатов
Примеры
Для примера представим, что всего в нашей выборке 25 результатов.
Мы хотим получить все результаты с ограничением по 10 результатов на страницу.

первые 10 результатов

"offset":0
}
JSON
Результат
{"result":[ /* первые 10 результатов */ ],"nextOffset":10,"total":25}
JSON
результаты: 10-20
Запрос
{
"limit":10,
"offset":10
}
JSON
Результат
{"result":[ /* Результаты с 11 по 20 */ ],"nextOffset":20,"total":25}
JSON
результаты: 20-25
Запрос

}
JSON
Пример SDK
Пример получения списка запросов из проекта %NN%:

<?php

$projectId = '%NN%'; // ID of your project
$limit = 10; // page size

$TVSession = new TV\Session();

$selectorKeywords = new TV\Pen($TVSession, 'get', 'keywords_2', 'keywords');
$selectorKeywords->setData(['project_id' => $projectId]);
$selectorKeywords->setLimit($limit);

// get all result pages
do{
	$page = $selectorKeywords->exec();
	if(is_null($page->getResult())) return var_dump($page->getErrors());

	// $page - array of keywords
	foreach($page->getResult() as $resultItem){
		var_dump($resultItem);
	}

	$nextOffset = $page->getNextOffset();
	if($nextOffset) $selectorKeywords->setOffset($nextOffset);
}while($nextOffset);

Фильтры (параметры filters и id)
Последнее обновление: 06.11.2020
Фильтры используются для получения, редактирования или удаления определенных объектов.
id используется для получения, редактирования или удаления конкретного объекта.

Структура параметра filters
filters представляет собой массив, содержащий объекты фильтров fieldFilter.
Фильтр имеет следующие характеристики:

fieldFilter.name - имя поля
fieldFilter.operator - оператор сравнения
fieldFilter.values - массив со значениями фильтра
fieldFilter.values - всегда массив, в том числе для операторов сравнения с одним значением (например, EQUALS)
{
"filters":[{
"name": /* Имя поля (строка) */,
"operator": /* Оператор сравнения (строка) */,
"values":[ /* Значения фильтра */ ]
}, /*...*/]
}
JSON
Значения фильтра
fieldFilter.values может содержать числа, строки или имена полей

Для указания поля в качестве значения фильтра, его необходимо указывать в таком формате: "Field:{Имя_поля}".
При сравнении с другим полем нельзя использовать операторы BETWEEN, REGEXP и NOT_REGEXP
Доступные операторы сравнения
Оператор	Описание сравнения	Функция
EQUALS	равно указанному значению	=
NOT_EQUALS	не равно указанному значению	!=
IN	есть в указанных значениях	IN()
NOT_IN	нет в указанных значениях	NOT IN()
GREATER_THAN	больше, чем указанное значение	>
GREATER_THAN_EQUALS	больше или равно указанному значению	>=
LESS_THAN	меньше, чем указанное значение	<
LESS_THAN_EQUALS	меньше или равно указанному значению	<=
BETWEEN	в промежутке между значениями 1 и 2. Последовательно можно перечислять несколько промежутков:

values = [10,20,50,100]; // 10..20 или 50..100
JSON
BETWEEN
STARTS_WITH	начинается с указанного значения	LIKE '%_'
CONTAINS	содержит подстроку с указанным значением	LIKE '%_%'
DOES_NOT_CONTAIN	не содержит подстроку с указанным значением	NOT LIKE '%_%'
REGEXP	удовлетворяет указанному регулярному выражению	REGEXP()
NOT_REGEXP	не удовлетворяет указанному регулярному выражению	NOT REGEXP()
IS_NULL	поле установлено в null	IS NULL
IS_NOT_NULL	поле не установлено в null	IS NOT null
* Регулярные выражения на wiki, пример записи:

^w+$
JSON

Сортировка результата (параметр orders)
Последнее обновление: 06.11.2020
Данный параметр предназначен только для операций чтения, т.е. для работы с оператором get.

Сортировка по умолчанию зависит от конкретного метода.

Структура параметра orders
orders представляет собой массив, содержащий объекты сортируемых полей fieldOrder.
Объект сортируемого поля имеет следующие характеристики:

fieldOrder.name - имя поля
fieldOrder.direction - направление сортировки (ASC или DESC)

Составные поля
Последнее обновление: 06.11.2020
Составное поле представляет собой такое поле, значение которого зависит от разных параметров.
Такие параметры принято называть определителями.

При выполнении запроса к API в качестве имени составного поля необходимо передавать помимо самого имени, значения его определителей, разделенных двоеточием ":".
Значения определителей должны идти в алфавитном порядке их имен.

Пример
Пример составного поля: частота ключевой фразы volume.
Данное поле присутствует в типе данных ключевое слово

Определителями поля volume являются следующие характеристики:

region_key - ключ региона
searcher_key - ключ ПС
type - номер формы частоты
Так будет выглядеть параметр fields для получения значения частоты в регионе Москва (213) в ПС Яндекс (0) в форме "!Ключевая !фраза" (3):

{
"fields":["volume:213:0:3"]
}
JSON
Так будет выглядеть параметр filters для получение ключевых фраз с такой же частотой, как в примере выше, но со значениями не менее 50:

{
"filters":[{
"name":"volume:213:0:3",
"operator":"GREATER_THAN_EQUALS",
"values":[50]
}]
}

Последнее обновление: 06.11.2020
Краткое описание
Инструменты сервиса Аудит предназначены для обнаружения и отслеживания ошибок или нежелательных состояний сайта.

Полученная информация может быть использована для исправления ошибок и оптимизации сайта.

Сервис Аудит включает в себя следующие модули:

Проверка индексации страниц

Карта сайта

Индексация
Последнее обновление: 06.11.2020
Краткое описание
С помощью этого модуля можно проверить индексацию страниц сайта в поисковых системах Яндекс, Google, Bing и Yahoo.

Информация о том, какие страницы еще не проиндексированы поисковыми системами, имеет важное значение для продвижения сайта, так как пока страницы не попали в базу поисковых систем, они не могут отображаться в выдаче. Особенно важно контролировать индексацию, когда на сайте обновляется содержимое страниц.

Для запуска проверки индексации необходимо предварительно добавить в проект ссылки на страницы сайта с флагом активности индексации с помощью методов add/urls_2 или add/urls_2/import.

Модуль Радар
Последнее обновление: 06.11.2020
Краткое описание
С помощью этого модуля можно отслеживать изменения на страницах сайта. В качестве отслеживаемых элементов используются:

заголовок страницы (тег title);
описание страницы (мета-тег description);
заголовок документа первого уровня (тег h1);
контент страницы (способ сканирования выбирается в методе audit_2/settings);
HTTP коды страниц.

Отслеживание содержимого заголовков, meta-описаний и текстов на страницах сайтов позволит заметить изменения и оперативно среагировать, например, в случае, если позиции сайта ухудшились из-за внесенных изменений.

Для запуска проверки необходимо предварительно добавить в проект ссылки на страницы сайта с флагом активности Радара с помощью методов add/urls_2 или add/urls_2/import.

Результаты проверок хранятся 180 дней.

Методы модуля Радар:
get - получить результаты проверки Радара
get: watcher/checker/price - получить стоимость проверки Радара
get: watcher/checker/status/percent - получить процент готовности проверки Радара
get: watcher/summary - получить сводку по результатам проверки Радара за выбранную дату
edit: watcher/checker/go - отправить проект на проверку содержимого страниц

Модуль Карта сайта
Последнее обновление: 06.11.2020
Краткое описание
С помощью этого модуля можно сгенерировать карту сайта (файл в формате XML). Карта сайта - это список всех страниц и разделов сайта, которые предназначены для индексации поисковыми системами.

Генератор карты сайта можно использовать, если по каким-то причинам система управления вашим сайтом не имеет встроенной возможности создать файл sitemap.xml с набором актуальных страниц сайта.

edit: sitemap/checker/go - запустить генерацию карты сайта в проекте.

Задачи (инструменты без создания проекта)
Последнее обновление: 03.01.2026
Некоторые инструменты можно использовать двумя способами: обычными методами, связанными с проектами, и через задачи. При работе через проект нужно создать проект, добавить в него поисковые системы и регионы, добавить поисковые запросы, запустить проверку и так далее — разными методами. С помощью задач же можно создать и настроить "проект" в одном запросе. После этого задача сразу отправится на проверку.

Работать обычными методами проектов удобнее, если вы регулярно работаете с одним и тем же проектом и используете разные инструменты. 

Работать через задачи удобнее, если вам нужно сделать однократную проверку и не требуется отслеживать прогресс в дальнейшем.

Что такое задача
Задача — это то же самое, что и проект, но с тремя отличиями:

После создания задачи выбранный инструмент запускается автоматически;

Задача не отображается в Списке проектов в интерфейсе и в результате работы метода get/projects_2/projects;

Задача автоматически удаляется через 24 часа после создания.

Доступные инструменты
